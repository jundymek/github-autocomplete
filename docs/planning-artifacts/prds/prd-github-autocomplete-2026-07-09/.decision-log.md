## Pre-made decisions (approved by Łukasz before this run)

- **D1 — Sandbox and reusability architecture.** Vite + React 19 + TS strict demo app (no Next.js, no monorepo). Reusability delivered via headless `useAutocomplete<T>` hook → generic `Autocomplete<T>` presentational component (props: `fetchSuggestions(query, signal)`, `renderItem`, `getItemKey`, `onSelect`) → thin `GitHubAutocomplete` adapter. Demo shows a second usage with a different data source as proof of reuse. PRD captures reusability as first-class requirements (FR-14, FR-17, NFR-5); the hook/component decomposition itself belongs to architecture.md.
- **D1a — Self-contained styling.** CSS Modules + design tokens as CSS custom properties (`--ac-*` with fallbacks); no Tailwind or CSS framework inside the component. Theming via custom-property overrides is part of the public API → FR-15.
- **D2 — GitHub access and the "50 per request" interpretation.** Client-side, unauthenticated by default; optional token raises limits (FR-16, NFR-6). Two parallel requests (`/search/users`, `/search/repositories`), `per_page=50` each → merge → case-insensitive alphabetical sort (localeCompare) → trim to 50 combined. Interpretation of the ambiguous brief bullet "limited to 50 per request" = 50 per API request AND max 50 combined displayed — documented explicitly in PRD §4.2 and Open Question 1. AbortController cancels stale requests (FR-3). 403 rate limit gets a dedicated readable state with retry-after info (FR-9).
- **D3 — Test pyramid as a requirement.** Vitest unit (sort/merge, mapping, limit), RTL+MSW integration (debounce, 3-char threshold, keyboard nav, all visual states, stale-request cancellation), thin Playwright e2e (new-tab via `context.waitForEvent('page')`, focus, axe; API mocked via `page.route`). GitHub Actions CI. Testing is an FR (FR-18, FR-19), not just process — the brief asks for a "meaningful snippet of your ability to test the code."
- **D4 — Dropdown, not modal; portal rendering.** WAI-ARIA combobox pattern (input `role=combobox` + popup `role=listbox`). Dropdown rendered via React portal to `document.body` so it works inside `overflow:hidden` containers/modals; absolute-positioning fallback is a documented trade-off if the portal proves too costly. → FR-7, NFR-1.
- **D5 — No pagination / infinite scroll.** "Limited to 50 per request" is a cap requirement, not a paging feature; paging would break "combined and displayed alphabetically" (GitHub sorts by score; full alphabetical order needs the whole set up front). Max-height + native scroll; no virtualization at 50 items. → Non-Goals §5, FR-7 consequence.
- **Debounce ~300 ms; sub-3-char queries fire no request, dropdown closed/idle.** → FR-1, FR-2.
- **A11y as NFR.** ARIA combobox semantics, keyboard support (↑/↓/Enter per brief; Escape added), visible focus, WCAG AA contrast (tokens finalized in design phase). → NFR-1, FR-12.
- **Also out of scope:** SSR, i18n, persistence, auth flows beyond optional token. → §6.2.
